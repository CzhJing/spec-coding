# 代码库探索（Codebase Exploration）

**触发时机：** 任何新功能、需求变更开始前，`/opsx:explore` 之前强制执行。

**目标：** 在开口问问题之前，先对现有代码建立真实认知，避免方案与现实脱节。

---

## 执行步骤（按序，不可跳过）

1. **定位相关模块**
   根据用户描述的需求关键词，找到最可能涉及的目录和文件。

2. **读取入口与接口**
   阅读相关模块的对外接口、路由定义、类型声明，理解现有约定。

3. **追踪数据模型**
   识别与需求相关的数据结构、数据库 schema 或状态定义。

4. **标记可复用逻辑**
   找出可以直接复用或需要扩展的现有函数 / 组件 / 服务。

5. **识别关键约束**
   记录会影响方案选择的硬约束（框架限制、性能敏感路径、安全边界等）。

6. **输出探索摘要**
   按下方格式呈现给用户，等待确认后再进入 `/opsx:explore`。

---

## 探索摘要格式

```
## 探索摘要

**相关文件：**
- `src/xxx.ts` — 现有 XXX 逻辑，涉及 YYY 接口
- `src/yyy/` — 负责 ZZZ 的模块

**现有约定：**
- 接口风格：REST / tRPC / GraphQL ...
- 错误处理：统一通过 xxx 抛出
- ...

**可复用：**
- `utils/xxx` 可直接复用于 ...

**关键约束：**
- XXX 模块不允许直接访问数据库，必须经由 YYY service
- ...

**探索结论：** <1-2 句话，总结对方案选择最重要的发现>
```

---

## 行为约束

- 探索阶段**只读**，不写任何代码、不修改任何文件
- 摘要呈现后，等待用户确认（"继续" 即可）再进入需求探索
- 若需求与现有代码差异极大（如需整体重构），必须在摘要中**明确指出**，不得隐藏
- 不能跳过此阶段直接进入 `/opsx:explore`
