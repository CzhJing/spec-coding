# 禁止行为清单

遇到决策分叉时先查此表。每条禁止行为都有对应的正确做法。

---

## 流程类

| 禁止行为 | 正确做法 |
|----------|----------|
| 跳过代码库探索直接进入需求探索 | 先探索代码库、输出摘要，用户确认后再执行 `/opsx:explore` |
| 没有 `/opsx:explore` 直接写代码 | 先 `/opsx:explore`，再 `/opsx:propose`，再执行 |
| 没有 OpenSpec 提案直接执行 | `/opsx:explore` 确认后必须走 `/opsx:propose` 锁定边界 |
| 执行超出 OpenSpec 边界 | 发现边界外需求，停下来更新提案，人工确认后再继续 |
| 跳过执行模式确认节点 | 分支创建后必须先向用户确认选择 Ruflo swarm 还是 Superpowers |
| 不归档变更 | 每次变更完成必须 `/opsx:archive`，规约必须保持最新 |

## 代码类

| 禁止行为 | 正确做法 |
|----------|----------|
| 一次性大面积修改 | 拆成 2-5 分钟原子任务，逐步提交 |
| 跳过 TDD | 严格 RED → GREEN → REFACTOR，测试通过才提交 |
| 跳过三级审查 | 实现、规约、代码质量三级全部通过才标记完成 |
| 审查失败静默重试超过 3 次 | 第 3 次失败必须触发升级机制，向用户上报，等待指示 |
| 前后端在同一会话并行 | 各自开独立会话，基于同一份 spec.md 并行工作 |

## 文件类

| 禁止行为 | 正确做法 |
|----------|----------|
| `plan.md` 散落在其他目录 | 固定写入 `openspec/changes/<name>/plan.md` |
| 靠猜测确定父分支 | 读 `git config --get branch.<current>.parent`，读不到就问用户 |
| 探索阶段写代码或修改文件 | 探索阶段只读，任何写操作都必须等到分支创建后 |
| 跳过 proposal 直接写 tasks | 制品依赖顺序：proposal → specs/design → tasks |
